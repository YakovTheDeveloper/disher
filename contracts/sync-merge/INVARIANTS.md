## И-1. `merge` идемпотентен ✅

`merge(S); merge(S)` == `merge(S)`.

Держится: `snapshot/index.ts:116` сравнивает **строгим** `>`. На втором прогоне у входящей строки
штамп ровно равен локальному (её же только что записали) → `>` ложно → записи нет. Тумбстоуны —
то же (`:97`).

Ассоциативность держится **при попарно различных штампах** (per-row `max` ассоциативен, а
tombstone-apply — монотонный фильтр поверх union). При равных штампах ломается — это следствие И-2,
а не отдельный инвариант.

## И-2. `merge` коммутативен, устройства сходятся ❌ НАРУШЕН

**Формулировка.** Порядок обмена не влияет на результат: если D0 и D1 обменялись состояниями, они
приходят к одному состоянию.

**Где ломается.** `snapshot/index.ts:116`:

```ts
if (!local || (inc.updated_at ?? '') > (local.updated_at ?? '')) {
```

Сравнение **строгое**. При `inc.updated_at === local.updated_at` и **разном содержимом** входящая
строка проигрывает — локальная сторона всегда выигрывает. Значит «кто мержит» определяет исход:
D0 оставляет своё, D1 оставляет своё. Ни один последующий merge их не сведёт (штампы равны навсегда).
Серверный блоб при этом флипается по последнему `push`. Тот же строгий `>` в тумбстоунах (`:97`).

**Почему это не «теоретическая» коллизия миллисекунд.** Штампы выдаёт не настенное время, а
монотонный счётчик (`write.ts:84`):

```ts
const next = Math.max(Date.now(), readHwm() + 1);
```

Как только **HWM обгоняет настенные часы**, `Date.now()` перестаёт участвовать вовсе, и `nextStamp()`
вырождается в детерминированный счётчик `hwm+1`, `hwm+2`, … **одинаковый на всех устройствах,
стартовавших с одного HWM**. А стартуют они с одного HWM регулярно: `observeStamp` (`:114`, `:95`)
поднимает часы каждого пира до максимума, увиденного в блобе.

**Сценарий отказа (воспроизведён исполнением).**

1. Устройство C с часами, спешащими на 5 минут, правит что угодно → штамп `12:05:00.000Z`
   (истинное время — `12:00`).
2. D0 и D1 (часы верные) синкаются → `observeStamp('12:05:00.000Z')` → у обоих `HWM = 12:05:00.000`.
3. Оба уходят офлайн. D0 правит `p1.name = 'A'`, D1 правит `p1.name = 'B'`.
4. У обоих `Date.now() ≈ 12:00` < `HWM+1` → **оба выдают ровно `12:05:00.001Z`**.
5. D0 мержит блоб D1: `'…001Z' > '…001Z'` → false → у D0 остаётся `'A'`.
   D1 мержит блоб D0: то же → у D1 остаётся `'B'`.
6. **Навсегда.** Разное имя продукта на телефоне и на ноутбуке; ни один синк это не лечит.

**Насколько узко окно (честно).** Нужно совпадение трёх условий: (а) общий HWM превышает настенные
часы обоих устройств в момент правки — размер окна равен величине рассинхрона часов; (б) обе правки
попадают в **одну строку**; (в) на **одном порядковом номере** записи после синка. Это сужает
вероятность, но **не разрывает цепочку**. Причём предусловие (а) достижимо и **без всякого
рассинхрона часов**: три `putRow` внутри одной миллисекунды уже оставляют `HWM − Date.now() = 2 мс`
(измерено).

**И-17 делает это на порядок хуже:** отравленный штамп переводит флот в вырожденный режим `hwm+1`
**навсегда**, а не на время рассинхрона. Тогда условие (а) выполняется всегда.

**Под KMP — обычное дело.** Клиент с секундной гранулярностью штампов (или любой, кто не
реализует HWM-счётчик буква-в-букву) плодит ties постоянно.

## И-3. Всякий наблюдённый штамп поднимает часы ⚠️ ДЫРА В `apply()`

**Формулировка.** Ни одна локальная запись не может получить штамп меньше уже увиденного —
иначе заведомо более поздняя правка тихо проиграет на merge.

**Держится** в `merge()`: `observeStamp` зовётся для каждого входящего `updated_at` (`:114`) и
каждого `deleted_at` (`:95`).

**Не держится** в `apply()` (`:69-79`): строки кладутся `bulkPut`'ом, `observeStamp` не зовётся ни
разу — ни для `updated_at`, ни для `deleted_at` тумбстоунов. HWM остаётся там, где был.

**Сценарий отказа.** Пользователь восстанавливает данные из файла («Загрузить из файла»,
`ProfileDrawer.tsx:124`), экспортированного с устройства, чьи часы спешили на час: в файле строка
`p1` со штампом `13:00`, настоящее время — `12:00`.

1. `apply(snap)` → `p1.updated_at = 13:00` в Dexie. **HWM не поднялся.**
2. Reload → `BackupGate` → `syncNow` → pull (у сервера `p1` нет или он старше) → merge (наблюдать
   нечего) → **push** → теперь `p1 @ 13:00` **на сервере**.
3. Пользователь правит `p1` → `nextStamp() = max(12:00, hwm+1) ≈ 12:00`.
4. Следующий `syncNow` → pull → `13:00 > 12:00` → **правка пользователя молча перезаписывается его
   же импортированной версией** и уезжает такой на сервер.

Воспроизведено исполнением: после правки в `p1.name` возвращается значение из файла.

**Важно (поймано адверсариальной проверкой):** вариант, где будущий штамп *уже* лежит на сервере в
момент импорта, **не срабатывает** — reload → `syncNow` → `merge` → `observeStamp` поднимет HWM
раньше, чем пользователь успеет что-то отредактировать. Дыра открывается только когда будущий штамп
попадает на сервер **пушем этого же устройства**: на свои собственные штампы `observeStamp` не
зовётся никогда.

## И-4. Лексикографическое сравнение ISO == хронологическое ⚠️ ДЕРЖИТСЯ ТОЛЬКО ДЛЯ JS

`merge` сравнивает штампы **как строки**. Это верно ровно для формата `Date.prototype.toISOString()`:
фиксированные `YYYY-MM-DDTHH:mm:ss.sssZ`, миллисекунды **всегда** три знака, даже нулевые.

Ломается на любом штампе без миллисекунд (проверено исполнением):

```
'2026-07-14T10:00:00.001Z' > '2026-07-14T10:00:00Z'   →  false   (лексикографически)
Date.parse(...001Z) > Date.parse(...00Z)              →  true    (хронологически)
```

`'.'` (0x2E) < `'Z'` (0x5A), поэтому строка с миллисекундами сортируется **раньше** строки без них,
хотя обозначает более позднее время. Порядок LWW инвертируется.

**Сегодня держится.** Проверено: другого источника штампов в проде нет — `nextStamp` (`write.ts:86`),
`created_at` через `new Date().toISOString()`, `stampImportRow` копирует `created_at`
(`index.ts:59-67`), Postgres `jsonb` сохраняет строки байт-в-байт (`backup.ts:18-23`).

**Сценарий отказа под KMP.** Kotlin `Instant.toString()` **опускает** нулевые миллисекунды
(`2026-07-14T10:00:00Z`). Примерно каждый тысячный штамп попадает ровно на границу секунды. Такая
строка проигрывает **любой** строке той же секунды с миллисекундами — включая заведомо более раннюю.
Правка, сделанная позже, теряется. Диагностируется как «иногда правки с телефона исчезают».

## И-5. Неизвестные таблицы блоба переживают round-trip ❌ НАРУШЕН — головное

**Формулировка.** Клиент, не знающий о таблице `X`, не должен уничтожать `X` в общем блобе.

**Механизма нет.** В блобе нет `schema_version`. Ни один из трёх слоёв не отвечает за сохранение
ключей, которых он не понимает:

1. `dump()` (`:47-55`) итерирует **`db.tables`** — локальную схему. У клиента со схемой v7 нет
   objectStore `insights` (добавлен в v8, `schema.ts:303`) → ключа `insights` в его блобе **нет
   вообще**.
2. `push()` → `PUT /api/backup` → `backup.ts:21`: `on conflict (user_id) do update set snapshot =
   excluded.snapshot` — **whole-blob replace**. Сервер по дизайну не знает схему пользовательских
   данных («The body is opaque jsonb», `backup.ts:9-10`) и не может сохранить ключ, которого ему не
   прислали. Он не мержит — он затирает.
3. `merge()` (`:104`) итерирует **`DOMAIN_TABLES`** — локальную константу. Входящие ключи не из
   списка не читаются вовсе. Старому клиенту чужие `insights` даже некуда положить.

**Сценарий отказа.** Пользователь сохранил 12 инсайтов (схема v8). Второе устройство работает на
старом билде со схемой v7.

1. Старый клиент: `pull` → в блобе `insights: [12 строк]`. `merge` их игнорирует.
2. `dump()` → ключа `insights` нет. `push` → серверный блоб **заменён без них**.
3. На v8-устройстве строки ещё живы локально и восстановятся на его следующем push (union по
   присутствию). Фактическая потеря наступает, когда **последнее v8-устройство теряет локальную
   копию**: переустановка, чистка сайта, `signOut`+`signIn` (зовёт `wipeLocalData`,
   `auth-store.ts:229`) или новое устройство. Единственная копия была на сервере — её стёрли.

**Сужение (найдено адверсариальной проверкой).** Устройство, которое **хоть раз открывало v8**, не
может тихо откатиться на v7: IndexedDB отвергает `open(name, 70)` при сохранённой версии 80 с
`VersionError`, и Dexie падает громко. То есть «планшет обновил базу, а потом закэшированный старый
бандл тихо снёс insights» **не происходит** — старый клиент должен быть таким, который v8 **никогда
не грузил**. Сценарий остаётся живым, но вектор уже, чем казалось.

**Под KMP — гарантированный:** нативный клиент по определению стартует с подмножеством таблиц.

**Что закрыло бы:** любое ОДНО из трёх — (а) `merge` протаскивает неизвестные ключи блоба в
holding-store и `dump` их возвращает; (б) сервер делает per-key union; (в) `schema_version` в блобе,
push от клиента ниже версии блоба отклоняется.

## И-6. Неизвестные колонки строки переживают merge ❌ НАРУШЕН

Строчный аналог И-5, и его легко проглядеть.

`merge` пишет входящую строку **целиком**: `await tbl.put(inc)` (`:117`) — IDB-replace всей записи,
а не слияние полей. Клиент, чья модель строки не содержит колонку `C`, отдаёт в блоб строку без `C`;
выиграв LWW, она затирает `C` у всех.

**Почему это молча.** `products.description` и `dish_items.details` добавлены **без бампа схемы**
(они неиндексированы), и мапперы подставляют `''`, если поля нет: `product/api/mappers.ts:22`
(`description: row.description ?? ''`), `dish/api/mappers.ts:13,26`, `schedule-food/api/mappers.ts:11`.
Потеря **не падает и не логируется** — она читается как «пользователь ничего не писал».

**Сценарий отказа.**

1. KMP-клиент моделирует `ProductRow` без `description` (его нет в индексах — о нём легко не узнать).
2. Пользователь на телефоне меняет вес продукта `p1` → строка уезжает в блоб без `description`.
3. Свежий штамп → выигрывает LWW у всех → `p1.description` = `undefined` везде.
4. Маппер рисует `''`. **Заметка пользователя исчезла молча.** Ни ошибки, ни тумбстоуна, ни следа.

Живого пути в сегодняшнем JS-флоте нет (`putRow` всегда получает полную строку; `updateRow` идёт
через `Table.update`, который **сливает** поля). Это чистый контракт для второго писателя.

## И-7. Всякое удаление оставляет tombstone в той же rw-tx ✅

Держится: `deleteRow` (`write.ts:143-153`) и `deleteRows` (`:158-172`) удаляют строку и пишут
`tombstones` в **одной** `db.transaction('rw', [table, 'tombstones'])`. Сырые записи забанены eslint
(`eslint.config.ts:94-107`) — у правила **два** селектора: прямые `db.<table>.<write>` и
коллекционные `.where(...).delete()`.

Прогон по всему `src/`: в прод-коде нет ни коллекционных `delete`, ни `.modify()`, ни `.clear()`
вне тестов, миграций и `wipeLocalData`. **Инвариант держится.**

**Дыры (все известные, ни одна не в проде):**

- Правило **не покрывает `.clear()` и `.modify()`** — их нет в селекторах. Будущий
  `db.products.clear()` или `.toCollection().modify(...)` пройдёт lint молча. Дыра на будущее.
- `auth-store.ts:19` (`wipeLocalData`) — массовый `.clear()` без тумбстоунов. **И это правильно**:
  при смене личности умирает вся база, а не отдельные строки.
- `e2e/bridge.ts:38-39` — `db.table(name).bulkAdd(rows)` в **динамической** форме, которую селектор
  статически поймать не может (признано в `eslint.config.ts:78-82`). Пишет неотштампованные строки.
  Только dev/e2e (`bridge.ts:14`).

## И-8. Всякая запись доменной строки штампует `updated_at` ✅ в проде

Единственный санкционированный путь — `putRow`/`putRows`/`updateRow`, все зовут `nextStamp()`.
Сырых `db.<table>.put/add/update` вне контракта в прод-коде нет.

Намеренные исключения: `merge()` пишет через `tbl.put` **напрямую**, чтобы сохранить чужой
`updated_at` (иначе `putRow` перештамповал бы его в `now()` и разрушил LWW — `write.ts:29-30`);
`apply()` штампует только legacy-строки (`updated_at ??= created_at`, `:59-67`).

Дыра: `bridge.bulkAdd` (dev, см. И-7).

## И-9. push только после pull+merge ✅ на клиенте, ❌ на сервере

**Клиент держит.** `syncNow` (`:209-224`) — единственный путь: pull → merge → prune → push, целиком
под Web Lock `'disher-sync'`, так что монтирование `BackupGate` и ручные кнопки не могут порвать
друг другу dump/push. Все прод-вызовы идут через одну обёртку `runSyncTracked`
(`shared/lib/sync/runSync.ts:33-54`).

**Сервер НЕ держит.** `PUT /api/backup` принимает блоб безусловно: нет `ETag`/`If-Match`, нет
версии, нет проверки, что писатель видел текущее состояние. Голый PUT **затирает всё**.

**Сценарий отказа.** Второй писатель реализует «сохранить» как прямой `PUT /api/backup` своим дампом,
не выполнив pull+merge, — потому что API этого не требует и ничем не намекает. Весь блоб заменяется
подмножеством, которое знает нативный клиент. Веб-данные исчезают одномоментно.

Сегодня доступно только через `window.__e2e.pushSnapshot` (`bridge.ts:27`, dev-only).
**Инвариант держится соглашением, а не контрактом.** Соглашение не переживёт второй кодовой базы.

## И-10. TTL тумбстоунов > максимального окна офлайна ⚠️ УСЛОВНО

`TOMBSTONE_TTL_MS = 90 дней` (`:140`). Инвариант **задокументирован** в коде (`:136-139`).

### (а) Часы вперёд — **транзиентный флап, а не перманентное воскрешение**

`pruneTombstones` (`:148-151`) считает cutoff **настенными часами** (`Date.now()`), а не монотонным
штампом. Устройство S с часами, спешащими на >90 дней, выпалывает **все** тумбстоуны и пушит блоб
без них.

Но воскрешения «у всех» **не происходит** — это моя первоначальная переоценка, снятая адверсариальной
проверкой (воспроизведено исполнением). `merge` **никогда не удаляет** локальный тумбстоун за
отсутствие во входящем блобе — цикл тумбстоунов (`:94-101`) только `put`'ает. А `pruneTombstones`
работает по **локальным** часам, поэтому здоровые устройства свои тумбстоуны сохраняют:

- S синкается → в блобе 0 тумбстоунов.
- Спящее Z (ещё держит строку локально) синкается → строка **воскресает в блобе**. ← транзиент
- Здоровое H синкается → у H тумбстоун жив → строка снова удалена, тумбстоун вернулся в блоб.
- Z синкается ещё раз → строка удалена и у Z.

Итог: **флап и схождение обратно к «удалено»**. Перманентное воскрешение требует, чтобы тумбстоун не
сохранился **ни на одном** устройстве — то есть чтобы удаление сделало само устройство с битыми
часами (или чтобы оно было единственным пиром).

### (б) Файловый экспорт — «спящее устройство» без верхней границы возраста ✅ сценарий держится

TTL ограничивает окно офлайна *устройства*, но JSON-файл экспорта — то же спящее устройство, только
вечное. Импорт годовалого файла (`apply`, `ProfileDrawer.tsx:124`) поднимает строки, удалённые 10
месяцев назад; их тумбстоуны выполоты с сервера ещё 7 месяцев назад. `merge` — union по присутствию
→ строки живы → `push` → **воскресли у всех**. Проверки возраста файла нет нигде.

## И-11. uuid уникален по ВСЕЙ базе, не по таблице ✅ держится — и он НЕСУЩИЙ

Тумбстоуны ключуются **одним `id`**: primary key `tombstones: 'id, table'` (`schema.ts:280`), и
`merge` строит `Map<string, TombstoneRow>` по `t.id` (`:92`), **не** по паре `(table, id)`. Поле
`table` едет как данные и решает только, к какой таблице применить **выживший** тумбстоун (`:123`).

Держится, потому что все id — `crypto.randomUUID()`. Это **контракт, а не следствие типов**.

**Насколько он несущий — выяснилось эмпирически.** Property-тест ассоциативности упал на первом же
прогоне, и контрпример был именно про это. Два тумбстоуна с одним `id`, но разными `table`, дерутся
за одну ячейку `Map`: побеждает более поздний `deleted_at` и **вытесняет** другой. Вытесненное
удаление теряется целиком — строка, которую оно покрывало, остаётся жить.

Последствие сильнее, чем «одно удаление не доехало»: **merge теряет и коммутативность, и
ассоциативность**. Один и тот же набор блобов даёт разный результат в зависимости от порядка:

```
local = { products: [x1 @ T0] }
A     = { tombstones: [{ id: 'x1', table: 'products',  deleted_at: T0 }] }   // удалить x1
B     = { tombstones: [{ id: 'x1', table: 'insights',  deleted_at: T8 }] }   // про другую таблицу

merge(local, A, B) → строки нет   (сначала удалили, потом вытеснили тумбстоун — поздно)
merge(local, B, A) → строка ЖИВА  (тумбстоун B занял ячейку; A старше → проиграл LWW → не применился)
```

Сегодня недостижимо (uuid не сталкиваются). Достижимо у второго писателя, который начнёт выдавать
id, уникальные только **внутри таблицы** (автоинкремент, композитный ключ). Реперный кейс:
`merge.props.test.ts` → «И-11: an id shared across tables breaks the tombstone track».

## И-12. Уникальность `(product_id, tag)` в `custom_tags` ❌ НАРУШЕН (но пользователь этого не видит)

**Дубли строк реальны.** Индекс — `custom_tags: 'id, product_id'` (`schema.ts:239`): primary key =
uuid, `product_id` — **обычный неуникальный** индекс, никакого `&[product_id+tag]`. Дедупликация
живёт только в писателе (`entities/custom-tag/api/mutations.ts:20-24`): прочитать теги этого продукта
**из локальной Dexie**, отфильтровать существующие, остальным выдать `crypto.randomUUID()`. Два
устройства независимо создают `{uuid-A, p1, 'варёное'}` и `{uuid-B, p1, 'варёное'}` — для LWW по
primary key это **две разные строки**, схлопнуть их нечему.

**Но impact мельче, чем кажется** (моя первоначальная оценка была неверна — опровергнуто
адверсариальной проверкой). `DetailsChips.tsx:74-79` дедупит чипы **по нормализованному тегу**:

```ts
for (const row of customTagRows) {
  const norm = normalizeTag(row.tag);
  if (!norm || seen.has(norm)) continue;   // ← дедуп
  seen.add(norm);
  customOut.push({ tag: row.tag, source: 'custom' });
}
```

а `upsertCustomTags` уже хранит нормализованную форму (`mutations.ts:16`). Двух одинаковых чипов
пользователь **не увидит**; `removeCustomTag` (`:51-56`) удаляет **все** совпадения. Реальная цена —
**невидимый мусор в блобе**, растущий с числом устройств.

**Зато достижимость шире, чем я думал.** `upsertCustomTags` читает `existing` (`:20`) и пишет (`:38`)
**вне транзакции**, а вызывающий `persistCustomTagsFromDetails` документирован как fire-and-forget.
Два перекрывающихся вызова для одного продукта и тега дают дубли **на одном устройстве** — второй
писатель для этого не нужен.

**Общая форма (важна для KMP):** LWW по primary key **не умеет** обеспечивать уникальность по любому
другому набору колонок. Любой natural key (`(product_id, tag)`, `(dish_id, product_id)`, …) держится
ровно до второго писателя — а при read-then-write вне транзакции не держится и до него.

В снапшот входят **только** 10 доменных таблиц Dexie + `tombstones` (`DOMAIN_TABLES`, `:29-40`).
**Намеренно вне** — и, следовательно, не переезжает между устройствами:

| JS-бандл | `catalog.json` (~715 системных продуктов) |

### Дыра: HWM часов течёт **между аккаунтами**

Изначально я записал это как «безвредно — часы двигаются только вперёд». **Это неверно.**
Пользователь B, входящий на общем устройстве, где у флота пользователя A был перекошенный (или

**протекает через границу аккаунта**.

(`auth-store.ts:315`). Для HWM такого сброса нет.

**Контракт для KMP:** свой персистентный HWM обязателен — без него И-2/И-3/И-17 разваливаются.

## И-14. 401 не убивает несинхронизированные правки 🟡 ПРИНЯТЫЙ TRADE-OFF

**Держится в главном.** 401 трактуется как **заявка**, а не приговор: `handleSessionExpired.ts`
фильтрует (только `kind==='auth'` && 401 && не ошибка формы логина), затем **переспрашивает** сервер
`probeSessionLiveness()`. `alive` → тост «попробуйте ещё раз». `unknown` (нет связи) → тост «данные
сохранены на устройстве». Wipe — **только** при подтверждённой смерти сессии.

**Остаточный trade-off (осознан, не баг).** При подтверждённо мёртвой сессии
(`handleSessionExpired.ts:94`) идёт `signOut({ skipFinalSync: true })` → `wipeLocalData()`. Финальный
синк намеренно пропущен (push с отозванной кукой всё равно даст 401). Значит **все правки с момента
последнего успешного синка гибнут — без вопроса пользователю**; wipe запускается fire-and-forget.

**Что усиливает риск.** Автосинка **нет**: только монтирование `BackupGate` + ручные кнопки. Ни
таймера, ни debounce, ни `beforeunload`, ни `visibilitychange`. Окно несинхронизированных правок =
вся сессия работы приложения.

**Стейл-комментарий:** `snapshot/index.ts:181` говорит про «401 (bearer expired mid-session)» —
bearer выпилен, сессия теперь httpOnly-кука.

## И-15. Whole-row LWW теряет конкурентные правки разных полей 🟡 ПРИНЯТО

Два устройства правят **разные поля одной строки** → выигрывает вся строка целиком, правка
проигравшего поля теряется. Осознанный trade-off (память `project_description_lww_tradeoff`).
**Не поднимать как баг.**

В реестре — потому что второй писатель обязан знать: единица разрешения конфликта — **строка**, а не
поле. Оптимистичный «патч одного поля» на KMP молча съест чужие правки соседних полей.

## И-16. `apply()` — деструктивный replace без валидации ⚠️

`apply(s)` (`:69-79`) в одной rw-tx **очищает все таблицы** (включая `tombstones`) и раскладывает
блоб. Проверок нет: ни схемы, ни версии, ни принадлежности пользователю. Путь пользователя
(`ProfileDrawer.tsx:121-128`): `pickJson()` (голый `JSON.parse`, `:51-66`) → `apply(snap as never)` →
`window.location.reload()`. **Ни confirm-диалога, ни бэкапа перед перезаписью.**

**Impact меньше, чем я написал сначала** (снято адверсариальной проверкой, воспроизведено
исполнением). Я утверждал, что стёртые тумбстоуны перестают распространяться. Это **неверно**:
порядок в `syncNow` — `pull → merge → prune → push` (`:214-217`), и `merge` **пере-записывает
каждый входящий тумбстоун локально** (`:99`) **до** того, как отработает `dump()`. Первый же синк
после импорта восстанавливает и тумбстоуны, и строки, которые сервер ещё держит (union по
присутствию).

**Что остаётся реальным:**

- Теряется **только то, что жило исключительно на этом устройстве и ни разу не было запушено** —
  то есть все правки с последнего успешного синка.
- **При синке ВЫКЛ спасать нечему:** `syncNow` выходит на `:211`, и `apply()` — окончательная,
  безвозвратная перезапись всей базы произвольным JSON-файлом.
- Обратная сторона, которую я не заметил: **при синке ВКЛ `apply()` — вообще не «восстановление»**.
  Импортированное состояние **объединяется** с серверным блобом (union + LWW), и пользователь
  получает **смесь**, а не содержимое файла. Кнопка называется «Загрузить из файла», а делает не то.
- `apply` — единственный путь в базу **в обход `observeStamp`** (см. И-3).

**✅ Закрыто частично (Корзина 1, 2026-07-15).** `handleImport` (`ProfileDrawer.tsx`) теперь
ставит два барьера ДО деструктивного `apply()`: (1) `isSnapshotShaped()` (`snapshot/index.ts`)
отвергает JSON, не похожий на бэкап Disher (объект, все значения — массивы, есть хоть один
известный ключ таблицы) → тостер, база не тронута; (2) `ConfirmModal` (danger-тон) — явное
подтверждение замены. Закрывает «выбрал не тот файл — снёс базу без спроса». **Остаётся** (не
Корзина 1): union-при-синке-ВКЛ (кнопка «Загрузить» при включённой синхре даёт смесь, а не
содержимое файла) и обход `observeStamp` в `apply()` — это семантика (Корзина 2), не гвард.

## И-17. HWM часов ограничен и восстановим ❌ НАРУШЕН

*(Пропущено в первой редакции; найдено адверсариальной проверкой.)*

**Формулировка.** Ни один входящий штамп не может необратимо увести часы устройства в будущее.

**Не держится.** `observeStamp` (`write.ts:73-77`) принимает **любой** результат `Date.parse`, лишь
бы он был конечным и больше текущего HWM. Верхней границы нет, clamp'а нет, пути сброса нет — HWM не
в снапшоте, не чистится `wipeLocalData`, не имеет потолка.

**Сценарий отказа.** В блоб попадает **одна** строка с `updated_at: '9999-12-31T23:59:59.999Z'`
(руками правленный файл экспорта → `apply()` → push; или любой второй писатель с багом в форматтере
даты).

1. Каждый пир на своём следующем `merge` зовёт `observeStamp` (`index.ts:114`) → **HWM = год 9999**
   в `localStorage`, навсегда.
2. С этого момента `nextStamp()` = `hwm+1` на **каждом** устройстве — `Date.now()` становится мёртвым
   кодом.
3. **Вырожденный режим И-2 становится постоянным рабочим режимом**, а не пятиминутным окном: любые
   две правки одной строки на разных устройствах, сделанные с одного и того же порядкового номера
   после синка, получают **идентичный** штамп → перманентная дивергенция.
4. Сбросить нельзя ничем, кроме ручной чистки `localStorage` **на каждом устройстве**.

Одна отравленная строка необратимо ломает LWW всего флота. Это делает И-2 материально хуже, чем
описано там.

**✅ Закрыто (Корзина 1, 2026-07-15).** `observeStamp` (`write.ts`) теперь отвергает любой
штамп больше `Date.now() + 24ч` (`OBSERVE_SKEW_CEILING_MS`) — потолок сильно выше реального
рассинхрона (NTP — секунды, ручной сдвиг — часы), но год 9999 режет наглухо. Клампится
только то, что **впитывается в HWM**; сама строка всё ещё мержится по своему `updated_at` —
мы просто не наследуем от неё безумные часы. Покрытие: `clock.test.ts` → «observeStamp()
refuses a far-future poison stamp». Остаточный вектор И-2 (равные штампы при рапид-райтах) —
это уже Корзина 2 (И-2 tie-break), сюда не входит.

## И-18. Битый блоб не может заклинить синк ❌ НАРУШЕН

*(Пропущено в первой редакции; найдено адверсариальной проверкой.)*

**Формулировка.** Некорректные данные в серверном блобе не должны делать синхронизацию
невосстановимой.

**Не держится, и это самое неприятное для KMP.** `merge()` пишет `tbl.put(inc)` (`:117`) — **любой
JSON без единой проверки**, на **каждом** синке. Сервер тоже ничего не валидирует
(`backup.ts:16-25` принимает произвольное тело).

**Сценарий отказа.** В блобе оказывается строка без `id` (или вообще не объект) — баг сериализации у
второго писателя.

1. `tbl.put(row)` бросает (IDB требует inbound key `id`).
2. Бросок внутри `db.transaction('rw', …)` → **вся rw-tx откатывается** → `merge()` реджектится.
3. `syncNow` реджектится **до** `push()` (`:217`) → **пуш не выполняется никогда**.
4. Значит **испорченный блоб невозможно заменить** — единственный код, который его перезаписывает,
   стоит после шага, который на нём падает.
5. Синк мёртв **на всех устройствах пользователя**, навсегда, без клиентского пути восстановления.
   `runSyncTracked` (`runSync.ts:44`) отправит ошибку в `handleSessionExpired` (не 401 → false) и
   молча проглотит: **онлайн-сбой не тостится**. Пользователь просто перестаёт синхронизироваться и
   **не узнаёт об этом**.

Сегодня недостижимо (единственный писатель — тот же код, что и читатель; только через
`window.__e2e.pushSnapshot`). **Достижимо в первый же день существования второго писателя.**

**✅ Закрыто (Корзина 1, 2026-07-15).** `merge()` (`snapshot/index.ts`) теперь скринит
каждую входящую запись через `usableId` / `usableTombstone` и **пропускает** те, у
которых нет строкового `id` (тумбстоуну нужен ещё строковый `deleted_at`), вместо того
чтобы дать `tbl.put` бросить и уронить rw-tx. Число отброшенных — в `console.warn` (не
молча). Покрытие: `merge.test.ts` → «И-18: a malformed incoming row cannot jam sync».

## И-19. Нет висячих ссылок ❌ не держится (мусор, не потеря)

*(Пропущено в первой редакции.)*

`deleteProducts` (`entities/product/api/mutations.ts:106`) **не каскадит** `custom_tags`. Строки с
`product_id`, указывающим на несуществующий продукт, не удаляются никогда и вечно ездят в блобе.

Не потеря данных — неограниченно растущий мусор + референциальный инвариант, который нигде не
заявлен. Второму писателю нужно знать, что каскады — **обязанность писателя** (`deleteRows` с
перечислением детей по внешнему ключу), а не свойство схемы.

## Что из этого — контракт для KMP-клиента

Минимальный набор, без которого второй писатель разрушает данные:

1. **Штампы — строки `YYYY-MM-DDTHH:mm:ss.sssZ`, три знака миллисекунд обязательны; сравнивать
   посимвольно, не парсить в instant** (И-4).
2. **Монотонные часы обязательны у каждого писателя:** персистентный HWM, `nextStamp() >
   max(Date.now(), HWM+1)`, `observeStamp()` на каждый чужой `updated_at`/`deleted_at` (И-2, И-3, И-13).
3. **Строка моделируется целиком**, включая неиндексированные поля (`description`, `details`), иначе
   merge затрёт их у всех (И-6).
4. **Никогда не push без предшествующего pull+merge** — сервер этого не проверяет (И-9).
5. **Удаление — только вместе с тумбстоуном в той же транзакции**; каскады — обязанность писателя
   (И-7, И-19).
6. **id уникален по всей базе, не по таблице** (И-11).
7. **Неизвестные ключи блоба нельзя терять** — до появления `schema_version`/passthrough клиент с
   подмножеством таблиц обязан протаскивать чужие ключи (И-5).
8. **Никогда не эмитить штамп из будущего** — одна такая строка необратимо ломает часы всего флота
   (И-17).
9. **Никогда не эмитить строку без `id`** — она заклинивает синк навсегда, без пути восстановления
   (И-18).
10. Равные штампы — **зона перманентной дивергенции**; порядок разрешения ties сегодня =
    «локальный побеждает» (И-2).

## Что осознанно НЕ инвариант

Не поднимать как баги — решения приняты:
- whole-row LWW вместо per-field (И-15).
- Гибель несинхронизированных правок при подтверждённо мёртвой сессии (И-14).
- Отсутствие автосинка (только mount + ручные кнопки).
- Отсутствие schema-версии на сервере — сервер намеренно «тупая полка» (но см. И-5: цена этого
  решения оплачивается клиентом, а клиент её не платит).
