3. **Ajv-дефолты Fastify — две мины, обе несущие:**
   - `coerceTypes: true` **ПЕРЕПИСЫВАЕТ тело** до хендлера ⇒ `Type.Integer()` на
     `amountKop` превратил бы строковый `"500"` в число и **зачислил бы деньги**
     там, где сегодня 400. Выключено в `buildApp`.
   - `removeAdditional: true` ⇒ `additionalProperties: false` **вырезает** лишнее
     поле, а не отклоняет. Поэтому тело `PUT /api/backup` = `Type.Unknown()`:
     любая схема со свойствами молча срезала бы из дневника юзера всё неназванное.
   - `allErrors: false` ⇒ в `fieldErrors` обычно ОДНА запись. Не «чинить»:
     Ajv сам не советует включать allErrors на недоверенном входе.
4. **Порядок хуков стал несущим.** Валидация схемы идёт МЕЖДУ `onRequest` и
   `preHandler`. `requireTrustedOrigin` был `preHandler` ⇒ оказался позади Ajv, и
   forged-запрос с битым телом получал 400 вместо 403. Переведён на `onRequest`
   (заодно не парсит 5MB тела). Поймал ратчет `csrf-origin-guard.test.ts`.
5. **`response`-схем нет намеренно.** Fastify ими СЕРИАЛИЗУЕТ и молча срезает

## 3. 🔴 Находка 1: спека зависит от `NODE_ENV`

`buildApp.ts:292` — `bug-reports`, `dev`, `diag-logs` регистрируются **только вне прода**:

```ts
if (process.env.NODE_ENV !== "production") {
  await app.register(bugReportRoutes, { prefix: "/api/bug-reports" });
  await app.register(devRoutes, { prefix: "/api/dev" });
  await app.register(diagLogsRoutes, { prefix: "/api/diag-logs" });
}
```

`@fastify/swagger` собирает спеку **из фактически зарегистрированных роутов**. Значит дамп,
снятый на дев-машине, включит 7 роутов, которых в проде нет.

**Следствие:** скрипт дампа обязан подниматься с `NODE_ENV=production`. Иначе спека врёт —
и врёт ровно тому потребителю (второму клиенту), ради которого её и заводят.

**Побочно:** это делает `/docs` в деве и дамп для клиента **двумя разными вещами**. `/docs`
пусть показывает то, что поднято локально (это полезно); коммитится — прод-дамп.

## 4. 🟡 Находка 2: схема перехватывает 400 и портит текст ошибки

Сегодня валидация живёт в хендлерах и шлёт рукописный `{ error: "text is required" }`.
С TypeBox+Ajv запрос отклоняется **до** хендлера. Дальше `setErrorHandler` → `toProblem` →
`hasStatusCode(err)` → `detail = err.message` = **Ajv-текст по-английски**
(`body must have required property 'text'`).

Это не гипотеза — автор уже налетал. `user-reports.ts:36`, дословно:

> *«Deliberately NO `minLength`/`maxLength` here: an empty/whitespace `text` is caught by the
> trim check below (so the `{error}` body the frontend reads is preserved)»*

### Поправка (2026-07-16, по коду): первая редакция §4 была неточна дважды

**(1) «Сообщения рукописные и русские» — неверно. Они английские.** Проверено грепом по
`status(400)`: `"text is required"`, `"text too long (max 2000 chars)"`, `"dishName is required"`,
`"invalid window dates"`. Пользователь и сегодня видит английский технический текст. Разрыв между
`"text is required"` и Ajv'шным `body must have required property 'text'` — узкий. **Косметика
ошибки НЕ является основанием для правил этой фазы.**

**(2) Настоящий риск — смена ПОВЕДЕНИЯ, а не текста.** `user-reports.ts:72` и `free-text-food`
намеренно **обрезают** переросший текст (`text.trim().slice(0, MAX_TEXT)`), а не отклоняют его.
Поставить `maxLength` в схему = превратить truncate в 400. Это регрессия для живого юзера,
пишущего длинный баг-репорт. Комментарий на `:36` предупреждает ровно об этом — просто он
объясняет через `{error}`-тело, а суть в поведении.

### 🔴 Поправка 2: `readApiError` понимают НЕ все фетчеры

Первая редакция утверждала «по форме фронт устойчив — `readApiError` понимает оба формата».
Устойчив тот, кто через него идёт. Проверено: **8 фетчеров идут, 2 — нет.**

| Фетчер | Читает | Судьба при problem+json |
|---|---|---|
| `features/feedback/api/submitUserReport.ts:25` | **только `body.error`** | `undefined` ⇒ юзер видит `HTTP 400` |
| `features/dev/bug-report/api/submitBugReport.ts` | только `body.error` | то же, но dev-only |
| остальные 8 (`runAnalysis`, `parseFreeTextFood`, `admin`, …) | `readApiError` | ок, оба формата |

`submitUserReport` — **прод, user-facing** (кнопка «Сообщить о проблеме» в настройках).
И он **уже сломан сегодня**: 401/403/500 идут через `setErrorHandler` ⇒ problem+json ⇒ он
показывает `HTTP 500`. Схема на роуте просто расширит уже существующую дыру на 400.

⇒ **Чинить переводом на `readApiError`, а не мостом `error`-поля в problem-теле.** Это
починка существующего бага, попутно снимающая блокер фазы. Легаси-ветку `{error}` в
`readApiError` при этом НЕ трогаем — рукописные `reply.status(400).send({error})` живы.

### Решение: дожать механизм, который уже заложен

`api/errors.ts` **уже** содержит `unprocessable(fieldErrors)` + `ProblemExtensions.fieldErrors`
(«Per-field validation messages (422)»), а `apiError.ts` **уже** их читает (`asStringRecord`).
Не шлёт их **никто**. Ajv отдаёт `err.validation[]` — массив `{ instancePath, message }` —
ровно то сырьё, ради которого поле заводили.

⇒ Ветка в `setErrorHandler` для `FST_ERR_VALIDATION`: разложить `err.validation` в
`fieldErrors`, вернуть человекочитаемый `detail`. **Статус оставить 400**, не 422 — 422 в
`CODES` есть, но менять статус существующим роутам = менять контракт ради красоты. Отдельное
решение, не в этой фазе.

### Правило для схем в этой фазе

Схема проверяет **форму**: типы, обязательность, `additionalProperties: false`.
**Границы и семантику** (`maxLength`, пустая-после-trim, кросс-полевые правила) хендлер
оставляет себе.

Обоснование — **поведенческое, не косметическое** (см. поправку 1 выше): там, где хендлер
намеренно обрезает переросший вход, `maxLength` в схеме поменяет truncate на 400. Схема
описывает, что запрос **разбираем**; хендлер решает, что с ним **делать**.

Следствие для спеки: она не будет документировать лимиты длины. Это честно — сервер их и не
применяет как отказ. Где лимит реально отказной, `maxLength` ставится осознанно, по одному.
