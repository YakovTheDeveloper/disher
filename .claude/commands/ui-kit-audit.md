# /ui-kit-audit — стюард дизайн-системы Disher

UiKitPage — это **не** обычный showcase, а живой источник правды о дизайн-системе со
встроенным аудитом: реестр `SECTIONS`, `ANATOMY` (разбор каждого примитива с
`critique` + `improve`) и `LOCAL_BUTTONS` (каталог долга). Беда в том, что часть
контента — **ручные снимки**: «Раскраска (факт)» в ANATOMY снята руками на дату из
комментария и НЕ авто-связана с источником. После серии мерджей витрина дрейфует от
реальности, накапливает повторы и застрявшие старые версии.

Этот скилл — **closed-loop стюард**: снять состояние → детектить дрейф (страница ↔
код) и несвязность (страница ↔ сама себя) → диагноз → вписать находки в страницу →
опц. точечно унифицировать. **Эталон правильности — двойной: (1) сама страница
должна быть внутренне связной** (нет повторов, нет протухших версий, общая картинка
консистентна), **(2) её содержимое должно соответствовать реальным компонентам.**

Целевые файлы (всё в `apps/food-calc/src/pages/ui-kit/`):
`UiKitPage.tsx` (реестры + рендер), `UiKitPage.module.scss`, `fixtures.ts`.
Прозовый источник правды дизайн-канона (для справки, НЕ переписывать): `tds/semantic-roles.md`,
`tds/visual-voice.md`, `tds/design-consistency-plan.md`, food-calc/CLAUDE.md (§ Design Philosophy).

Если дан аргумент (`$ARGUMENTS`) — сузь весь проход до этой секции/компонента, но
шаги те же. Без аргумента — вся страница.

Делай строго по фазам. Не пропускай фазы 1–2 ради скорости — дрейф ловится именно там.

## Фаза 0 — Снять инвентарь страницы

1. Прочитай `UiKitPage.tsx` ЦЕЛИКОМ (файл большой, >1600 строк — дочитай все
   страницы Read), плюс `UiKitPage.module.scss` и `fixtures.ts`.
2. Построй у себя инвентарь (в голове/черновике, не в файл):
   - **Импорты компонентов** — что витрина показывает и откуда.
   - **`SECTIONS`** — реестр секций (id/index/title).
   - **`<Specimen>`** — каждый живой образец: какой компонент, какие пропсы, в какой секции.
   - **`ANATOMY`** — каждая запись: `id`, `title`, `parts[]` (part·role·**css «факт»**·why·`smell?`·`preview?`), `critique`, `improve`, `win?`.
   - **`LOCAL_BUTTONS`** — каждая строка долга: `cls`, `where`, `does`, `shouldBe`, `preview`.
   - Дату из комментария-снимка ANATOMY («Raskraska = hand snapshot of the CSS on `<дата>`»).

## Фаза 1 — Сверка с реальным кодом (страница ↔ код)

Для КАЖДОГО пункта инвентаря проверь соответствие реальности:

3. **Импорты / Specimen'ы.** Каждый импортируемый компонент ещё существует по
   своему пути? Проп, который дёргает Specimen, ещё есть в API компонента? Если
   импорт битый или проп мёртв — это дрейф (превью молча теряет стиль или падает сборка).
4. **Новые shared-примитивы, которых нет в ките.** Пройди по `shared/ui/**`
   (атомы, Typography, Button-семейство, input, overlay-каркасы) и сравни со списком
   импортов витрины. Заметный переиспользуемый примитив, которого НЕТ ни в одном
   Specimen, — кандидат на добавление (новая секция/образец).
5. **ANATOMY «Раскраска (факт)».** Для каждой `part` с `css` открой реальный
   `.module.scss` компонента и сверь снимок со строкой. Где снимок разошёлся с кодом
   (другой радиус, цвет, вес, токен; вариант, которого уже нет; проп, который сняли) —
   зафиксируй точную правку. Здесь же проверь `critique`/`improve`: не описывают ли
   они **уже сделанную** работу (напр. «перевести NumberInput на --field-*», когда
   он уже переведён) — такой текст протух.
6. **LOCAL_BUTTONS (каталог долга).** Для каждой `cls`:
   - Грепни класс по app. Класс ещё существует? Уже унифицирован (`shouldBe` стоит
     «✓ объединено», но строка всё ещё подана как активный долг)? Тогда долг закрыт.
   - Скан на **новые** одноразовые: `\.\w*Btn`, `\.\w*Button`, `\.fieldLabel`,
     `\.sectionLabel`, `\.closeButton`, локальные `.xxxBtn`-классы в `features/`,
     которых нет в каталоге, — занеси кандидатами в долг с предложенным `shouldBe`.

## Фаза 2 — Сверка страницы самой с собой (страница ↔ страница)

Это ваш главный критерий — внутренняя связность. Проверь:

7. **Дубли и застрявшие версии.** Один и тот же компонент показан в двух секциях /
   двумя Specimen'ами без причины. Старая версия описания/образца, оставшаяся рядом
   с новой. Две записи ANATOMY, противоречащие друг другу.
8. **Протухшие счётчики и утверждения.** Числа в тексте («было 11 вариантов → 4»,
   «~11 мест рисуют свой .fieldLabel», «~40+ кнопок», «снесены 7 мёртвых») —
   пере-выведи реальное число грепом и поправь. Расхождение цифры с кодом = находка.
9. **Мёртвые флаги.** `smell: true` на `part`, чей запах уже устранён (по Фазе 1);
   `win: true` на сущности, которая перестала быть консистентной; `desc`/`intro`
   секции, описывающие снятую механику.
10. **SECTIONS ↔ рендер.** Реестр секций совпадает с фактически отрендеренными
    `<Section>` (id/порядок)? Scroll-spy и nav читают `SECTIONS` — рассинхрон ломает навигацию.

## Фаза 3 — Системные предложения по унификации (движок советов)

Не пофайлово, а поперёк инвентаря — ищи системные запахи:

11. **Одна семантика — N примитивов.** Одна роль («выбери один из N», «вторичная
    CTA», «назад», «закрыть ×», «метка поля», «заголовок») нарисована несколькими
    разными примитивами/классами. Это корень ощущения «бардака». Предложи ЛИБО один
    канон, ЛИБО явное разведение ролей — и зафиксируй какую где (как уже сделано для
    Tabs «шаги визарда» ↔ NavSwitcher «разделы дома»).
12. **Мёртвые варианты/пропсы.** Вариант компонента или декоративный проп с НУЛЁМ
    боевых использований (как снесённые secondary/tertiary/danger у Button, как
    снятые underline/color у NumberInput) — предложи culling. Грепни использования прежде чем звать мёртвым.
13. **Токен-дрейф.** Компонент рисует сырые литералы (`#ccc`, `#0070f3`, `14px`)
    вместо системных токенов (`--field-*`, `--heading-*`, `--color-*`, `--space-*`).
    Сверься с token-гейтом (stylelint, `stylelint-legacy-baseline.cjs`) — новые
    нарушения вне baseline особенно важны.
14. **Свой каркас вместо канона.** Компонент руками повторяет stripe-fork / warm-row /
    card-palette вместо общего mixin/примитива (напр. HypothesisListItem копирует
    stripe-fork). Предложи вынести в один примитив.

Каждое предложение формулируй КОНКРЕТНО (что именно свести к чему), чтобы оно стало
готовой `ANATOMY.critique` + `ANATOMY.improve` или строкой `LOCAL_BUTTONS`.

## Фаза 4 — Вписать находки ПРЯМО в страницу

Результат живёт в самой витрине — отдельный отчётный md НЕ создавать.

15. Внеси правки в `UiKitPage.tsx` (и при нужде `.module.scss`/`fixtures.ts`):
    - **Дрейф снимков** — обнови `css` в `ANATOMY.parts` под реальный CSS; пере-стампи
      дату в комментарии-снимке («hand snapshot ... on `<сегодня>`») если трогал снимки.
    - **Связность** — убери дубли, поправь счётчики, сними мёртвые `smell`/`win`,
      обнови протухшие `critique`/`improve`/`intro`/`desc`.
    - **Новое** — добавь Specimen для незанесённого примитива (помни: pages могут
      импортировать widgets/features вниз по FSD — это легально для dev-страницы);
      заведи новую `ANATOMY`-запись или `LOCAL_BUTTONS`-строку.
    - **Долг** — закрытые позиции помечай «✓ объединено …» (как существующие
      примеры), новые — добавляй с `preview` живого класса.
    - **Системные советы** — впиши как `critique`/`improve` соответствующей сущности.
16. Держись стиля файла: тот же тон комментариев, та же структура записей, русский
    текст находок, ASCII-разделители секций как в файле. Не плодить новые
    `.*Label`/`.xxxBtn` — переиспользуй существующие примитивы и токены.

## Фаза 5 — Точечные рефакторы в app (опционально, по подтверждению)

Только если в Фазе 3 нашлись механические унификации (свести `.xxxBtn` к Button-варианту, `.backBtn`-места к `BackButton`, close-× к одному примитиву).

17. Покажи юзеру список кандидатов на рефактор (что→к чему, сколько мест) и спроси,
    какие делать. НЕ начинай рефактор сам без подтверждения по каждому.
18. Делай по ОДНОМУ за раз. После каждого: прогони relevant typecheck + eslint +
    stylelint на затронутые файлы. На красном — откати ИМЕННО этот рефактор и сообщи,
    не наслаивай второй на непроверенный первый.
19. После успешного рефактора обнови соответствующую `LOCAL_BUTTONS`-строку на
    «✓ объединено …».

## Жёсткие правила (git-safety + честность)

- **Только working tree.** КАТЕГОРИЧЕСКИ НЕЛЬЗЯ: `git commit`, `git push`,
  `git stash`, двигать/откатывать чужие uncommitted-изменения. В проекте параллельные
  сессии; коммитит только юзер по явной просьбе.
- **Scope.** Трогай UiKitPage-файлы + (в Фазе 5, по подтверждению) конкретные файлы
  рефактора. Чужое не трогай.
- **Дизайн-канон.** Любые правки — через токены и существующие примитивы
  (stripe-fork / warm-row / ambient / `--field-*` / `--heading-*`). Не вшивать
  `display` в shared-mixin'ы. Канон — food-calc/CLAUDE.md § Design Philosophy.
- **Верификация перед «готово».** Прогони typecheck + eslint + `lint:style` на
  затронутом коде. Витрина рендерится в браузере (`npm run dev-local` → роут ui-kit) —
  если не открывал браузер, честно скажи, что browser-smoke не делал.
- **Отчёт в конце прохода.** Кратко: что было дрейфом (снимки/счётчики/долг), что
  починил в странице, какие системные предложения вписал, какие рефакторы предложены
  и ждут подтверждения. Без воды.
